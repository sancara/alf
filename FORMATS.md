# alf — formats & contracts

Reference for the data contracts the CLI implements. Two principles run through all of it:

- **Intent vs fact** (like `Cargo.toml` / `Cargo.lock`): the *manifest* declares what the project wants (loose); the *lock* records what was resolved (exact). The lock is generated, never hand-edited.
- **Committed vs machine-local**: anything that should travel with the repo and apply to everyone is committed (`AGENTS.md`, `.alf/manifest.toml`, `.alf/lock.toml`, `.alf/config.toml`). Anything that's about *this developer on this machine* lives in `~/.config/alf/` and is never committed.

`alf` never calls a model, so it stores **no API keys and no provider settings**. Provider auth belongs to whatever agent tool reads the skills.

Skills are **flat** in v1: a skill never depends on another skill.

---

## 1. Catalog repo layout

The catalog is a git repo (remote = the developer's own). It is the single source of truth for skills.

```
catalog/
  skills/
    understand-the-problem/
      SKILL.md            # frontmatter: name, version (semver), description
      resources/          # optional, copied alongside the skill
    execution-plan/
      SKILL.md
    quality-reviewer/
      SKILL.md
    frontend-expert/
    backend-expert/
    devops-expert/
    security-expert/
  README.md
```

**Resolution is filesystem + frontmatter — no index file.** `alf` discovers skills by scanning `skills/*/SKILL.md` and reading the `version` from each frontmatter. The directory *is* the index; an extra catalog index would just be a second source of truth to drift. `alf list` works by scanning.

A skill's version is whatever its frontmatter says at the catalog commit being used. By default `alf` uses the catalog clone's current `HEAD`; a project may optionally pin a specific ref (see §4). Old versions are not re-fetched on demand — reproducibility comes from the lock plus the copied files, not from re-resolving history.

## 2. `.alf/manifest.toml` (committed — the intent)

Declares which skills this project uses. Hand-editable; `alf add` / `alf init` write to it.

```toml
# Skills this project uses. Version requirements are semver, like Cargo.
# "*" means "track the latest in the catalog".
[skills]
understand-the-problem = "*"
execution-plan         = "*"
quality-reviewer       = "*"
backend-expert         = "*"
devops-expert          = "*"
```

Requirements are intentionally loose (`"*"`, or `"^0.2"`, `"=0.2.0"` if a project wants to hold a line). The exact version actually installed is pinned in the lock, not here.

---

## 3. `.alf/lock.toml` (committed — the fact, generated)

Records exactly what was resolved and copied in. Generated by `alf`; not hand-edited.

```toml
# Generated by alf. Do not edit by hand.
lock_version = 1

[catalog]
remote = "git@github.com:dev/alf-catalog.git"
commit = "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0"  # resolved against this commit

[[skill]]
name    = "understand-the-problem"
version = "0.2.0"
hash    = "sha256:9f2c…"          # content hash of the copied SKILL.md (+ resources)
source  = "skills/understand-the-problem"

[[skill]]
name    = "execution-plan"
version = "0.2.0"
hash    = "sha256:1a7b…"
source  = "skills/execution-plan"

[[skill]]
name    = "quality-reviewer"
version = "0.1.0"
hash    = "sha256:c4d8…"
source  = "skills/quality-reviewer"
```

Two things the `hash` buys you:

- **Drift / `glearn` signal.** If a copied skill's current content no longer matches its locked `hash`, it has local modifications — exactly the candidate to promote upstream with `glearn`.
- **Tamper detection.** A skill changing out from under the lock (supply-chain tampering) is visible, which matters because instruction files are an attack surface.

Only **catalog-sourced** skills appear in the lock. Skills created locally in the repo (e.g. project-specific learnings that never came from the catalog) are just files under `skills/`, tracked by git, not by the lock.

## 4. `.alf/config.toml` (committed — project behavior)

Per-repo `alf` behavior. Committed, so it applies to everyone working in the repo. This is the file `AGENTS.md` points to for the stop-rule threshold.

```toml
[loop]
stop_after_attempts = 3       # execution-plan stop rule for this repo

[catalog]
# Optional. Pin resolution to a known catalog commit for team reproducibility.
# Omit to track the catalog's default branch.
# ref = "a1b2c3d4"
```

Note what is **not** here: the agent *target/profile* (claude, gemini, …) is a per-developer choice, not a repo property, so it lives in the machine config (§5), never committed. Committing a target would force one tool on everyone who clones — the opposite of the portability we want.

---

## 5. `~/.config/alf/config.toml` (machine-local — never committed)

Everything that's about this developer on this machine.

```toml
[catalog]
remote = "git@github.com:dev/alf-catalog.git"
path   = "~/.config/alf/catalog"   # local clone

[defaults]
target = "claude"                  # optional tool-specific extras only; see §6
```

That's the whole machine config: where the catalog lives, plus an optional default target for tool-specific extras (§6). No keys, no provider, no model — `alf` doesn't talk to any of them.

---

## 6. Project scaffold & tool targets

What `alf init` writes into a repo. Everything here is **committed**, so a clone is up and running with zero post-clone steps — any OS, any of the supported tools.

```
my-repo/
  AGENTS.md              canonical instructions — read by all four tools
  CLAUDE.md              one-line pointer to AGENTS.md (Claude Code's native file)
  .agents/skills/        canonical skills — read by Antigravity, Cursor, Copilot
    understand-the-problem/SKILL.md
    …
  .claude/skills/        mirror of .agents/skills/ — read by Claude Code
    understand-the-problem/SKILL.md
    …
  .alf/
    manifest.toml
    lock.toml
    config.toml
```

**Where each tool reads** (verified mid-2026):

- `AGENTS.md` (project root) — all four natively. Antigravity needs ≥ v1.20.3.
- `.agents/skills/` — Antigravity, Cursor, Copilot.
- `.claude/skills/` — Claude Code. (Cursor and Copilot read it too, but `.agents/skills/` already covers them.)

**Canonical vs mirror.** `.agents/skills/` is the single source of truth: it's what the lock tracks and what `plearn` edits. `.claude/skills/` is a mirror that `alf` regenerates on every `init` / `add` / `update` / `plearn`. The developer never touches or thinks about the mirror.

**No symlinks.** Skills are copied into both directories, not linked. Copies survive a Windows clone where git may not materialize symlinks, and they keep the repo self-contained and diffable. The duplication is a handful of small text files. (This is why OS detection proved unnecessary: dropping symlinks dropped all per-platform logic with it.)

**Uniform scaffold.** Because all four current targets read these standard locations, `alf init` writes the same set for everyone — there's no per-developer branching to materialize, and no post-clone step. The machine-config `target` (§5) is reserved for optional tool-specific extras (e.g. a `GEMINI.md` pointer for Antigravity older than v1.20.3) and for future tools whose layouts genuinely conflict; it does not change the core scaffold above.

---

## What's deliberately absent (v1)

- **No catalog index file** — scanning + frontmatter is the single source of truth.
- **No provider / API-key config** — `alf` never calls a model.
- **No skill dependencies** — skills are flat; revisit only if a real need appears.
- **No re-fetch of old versions** — the project is self-contained (copied skills + lock); reproducibility doesn't require re-resolving the catalog's history.
- **No symlinks and no per-OS logic** — skills are copied, so nothing breaks on a Windows clone and there's no platform branching to maintain.
